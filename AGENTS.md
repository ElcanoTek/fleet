# AGENTS.md

Operating guide for AI coding agents (Claude Code, Codex, Cursor, opencode, Goose,
Gemini CLI, ‚Ä¶) working in the **fleet** repository. It follows the
[agents.md](https://agents.md) convention; `CLAUDE.md` is a symlink to this file so
Claude Code and any `AGENTS.md`-aware tool read the same instructions.

Humans, start with [`README.md`](README.md) and [`CONTRIBUTING.md`](CONTRIBUTING.md);
this file is the agent-facing distillation, not a replacement for them.

## What fleet is (one paragraph)

fleet is a self-hosted, general-purpose agent platform. **One** Go process runs
interactive chat *and* a scheduling engine on one box, driven by **one** unified
agent runtime (`internal/agentcore`). Every agent tool call ‚Äî bash, Python, file
I/O, MCP ‚Äî executes inside a rootless-Podman sandbox; tools and data are reached
through an MCP catalog whose credentials are brokered host-side. See
the README "Architecture at a glance" for the full picture.

## Build ¬∑ test ¬∑ lint (run before opening any PR)

```sh
make build        # compile-check ./... AND emit ./fleet + ./fleet-admin
make compile      # go build ./...   (compile-check only; no artifacts)
make test         # go test -p 1 ./...   ‚Äî run in the FOREGROUND
make test-race    # go test -race -p 1 ./...   (use when touching concurrency)
make test-cover   # run Go tests with coverage profiling (writes coverage.out)
make lint         # golangci-lint run    ‚Äî the lint gate; must pass clean
make fmt          # gofmt -w .
make tidy         # go mod tidy
```

When you touch `web/` (the Next.js app):

```sh
cd web && npm ci && npm run lint && npm run test && npm run build
cd web && npx playwright test --project=mocked     # mocked e2e
```

CI mirrors all of this ‚Äî Go build/vet/lint/test (including a `-race` lane) plus a
`govulncheck` dependency-CVE scan, a Grype container-image CVE scan (fail on a
fixable CRITICAL) of the sandbox image, web lint/test/build, Playwright (mocked
**and** live, against a real backend + sandbox), and a gitleaks secret scan. **Every job
must be green before merge.** Tests are deterministic without a
live model: use the fake-LLM seam (`internal/fakellm` via `OPENROUTER_BASE_URL`),
never a real key.

## Non-negotiable invariants ‚Äî do NOT weaken these

These are the security and design guarantees the whole project rests on. A change
that breaks one is wrong even if tests pass. The *why* behind several of them is
recorded as Architecture Decision Records in [`docs/adr/`](docs/adr/) ‚Äî a change
that adds, weakens, or reverses an invariant must add or supersede an ADR in the
same PR.

- **The sandbox is mandatory.** The agent loop runs in the fleet process, but
  every agent tool call (bash, Python, file I/O, MCP) runs inside the
  rootless-Podman sandbox ‚Äî there is **no** fast path that skips it, and the host
  enforces all policy. The loop holds no privileged local executor of its own:
  each tool call is handed to the sandbox under host policy. MCP credentials are
  brokered **out-of-process** (issue #167) and **never** enter the sandbox ‚Äî the
  broker injects them only when it runs a delegated MCP call host-side.
- **Credentials stay host-side.** MCP/connector credentials are brokered on the
  host and **never** enter the sandbox, the agent container, the model context, or
  logs. Never ship a secret into a container or print one.
- **Governance is one core.** `agentcore.Run` is the single governed loop (policy,
  cost/token ceilings, audit, notes). New entrypoints **adapt I/O around it** ‚Äî
  they must not fork a second, weaker governance path.
- **No secrets in the repo.** gitleaks gates CI. Use the fake-LLM seam and obvious
  placeholders in tests; the real `OPENROUTER_API_KEY` lives outside the repo.
- **Honesty in docs.** Claim only shipped, tested capabilities. (The README
  Standards section lists MCP and Agent Skills as shipped.) If you add a
  capability, document what it actually does ‚Äî and what it does not. (Example: a
  skill's `allowed-tools` frontmatter is parsed but **not** enforced as a hard
  authorization gate ‚Äî the docs say so plainly rather than implying a boundary
  that isn't there.)
- **Client content is external.** Branding, the MCP catalog, personas, protocols,
  skills, and the sandbox Containerfile live in an out-of-repo client-config
  bundle (`FLEET_CLIENT_CONFIG_DIR`). fleet ships only the generic `config/default`
  bundle ‚Äî do not add client-specific content to this repo.

## Conventions

- Single Go module `github.com/ElcanoTek/fleet`, Go 1.26. Keep it `go vet`- and
  `golangci-lint`-clean ‚Äî lint failures block CI.
- **Go Test Coverage**: CI runs with `-coverprofile=coverage.out -covermode=atomic` on the plain `go test` step. Local coverage can be run using `make test-cover`. Project coverage must not drop more than 2% compared to the base branch, and new patch code in PRs must have at least 60% coverage, configured via `codecov.yml` at the repo root.
- **Match the surrounding code:** naming, idioms, and comment density. The
  `internal/agentcore` package comments explain *why* each governance invariant
  holds ‚Äî preserve that level of explanation when you extend it.
- Run tests in the **foreground**. Do not background `go test`, and do not
  `pkill -f 'go test'` (it can kill the shell). Prefer `make test` (it sets
  `-p 1`, which the suite expects).
- **Composer & Textarea heights**: In `web/src/app/chat/ui/Composer.tsx` and `chat-experience.tsx`, avoid mixing JS-based auto-grow style height clamping with Tailwind `max-h-*` CSS classes on the composer textarea, to prevent layout flickering. Use `MAX_COMPOSER_HEIGHT_PX` (200px) as the single source of truth.
- **Composer keybindings & preferences**: The user's send preference (`fleet.sendKey` in localStorage) controls whether Enter or Ctrl+Enter (with Cmd+Enter) sends a message. The keydown handler adjusts based on this config and skips sending on touch devices to let mobile keyboards use native enter keys.
- **Bulk Conversation Operations**:
  - **Soft Delete**: Setting `FLEET_CONVERSATION_SOFT_DELETE=true` switches deletes to soft-deletes (`deleted_at` timestamp). A 30-day background sweeper permanently purges soft-deleted rows. Soft-deleted conversations are excluded from retrieval and search query filters.
  - **Bulk APIs**: Bulk delete and patch actions are rate-limited to 100 IDs per request. Transactional pre-checks protect ownership; any single foreign ID fails the entire request (403 forbidden) to ensure consistency.
  - **Bulk UI**: Uses checkboxes for selection, warning users when selecting more than 50 conversations. The confirmation modal enforces a 3-second disabled countdown for delete actions.
- **Conditional Task Execution (run_if)**:
  - Pre-run shell gates (`run_if`) are evaluated serially on the host as the fleet process user prior to task promotion.
  - The evaluation is restricted to `PATH=/usr/bin:/bin` and `HOME=/tmp` with a custom timeout.
  - If a gate skips a recurring task, its status stays `scheduled`, and `scheduled_for` is advanced to the next cron occurrence. For one-shot tasks, status remains `scheduled` but the time is not advanced, acting as a soft hold.
- **Batch Task Operations**:
  - **Batch APIs**: `POST /tasks/batch` allows batch task creation of up to 100 tasks. In atomic mode (`atomic: true`), all tasks are validated up front and created in a single DB transaction (returning `422 Unprocessable Entity` with errors if any fail). In non-atomic mode (`atomic: false`), it behaves best-effort and returns `207 Multi-Status` for partial success.
  - **Rate Limiting**: Rate limiter consumes `N` tokens for `N` tasks in a batch (instead of 1 token per batch request) via `apikeys.Manager.ConsumeN`.
  - **Single Multi-row Insert**: `db.AddTaskBatch` inserts multiple tasks in a single query via parameterized multi-row insert rather than individual sequential inserts.
  - **CLI**: `fleet task batch-create --from-file tasks.json [--atomic]` allows creating tasks from a JSON file (or stdin via `-`).
- **Task Definition Import/Export**:
  - **Task Names**: Scheduled tasks now support an optional `name` field (mapped to the database `tasks.name` column with a partial unique index). Empty/empty-string names represent "unnamed" tasks and are always created fresh on import. Non-empty names are unique and serve as conflict keys.
  - **Payload Limits**: Imports are rate-limited/capped to at most 100 task records per request (matching bulk API policies). Payload-internal duplicate name entries are validation errors.
  - **Conflict Behaviors**: Mode `conflict=error` aborts the batch on any collision; `conflict=skip` skips colliding entries and writes others; `conflict=replace` performs an in-place update of colliding entries. Mode `conflict=replace` requires admin permission.
  - **Formats**: Support both JSON and YAML envelopes (via `github.com/goccy/go-yaml`). Version is set to `"1"`.
- **Read-only conversation sharing (#226)**:
  - **Opt-in public links**: `POST /conversations/{id}/share` mints a 256-bit `crypto/rand` (base64url) token stored in `conversations.share_token`; `DELETE` revokes it. `GET /shared/{token}` returns a read-only snapshot (title/model/messages) that **deliberately omits the conversation id and owner email**. Optional `share_expires_at` is enforced server-side.
  - **Public-but-proxied**: `/shared/{token}` is token-only-gated (shared secret ‚Äî only the Next proxy reaches it, not the open internet) and IDENTITY-less; the share token is the authorization, token entropy is the confidentiality guarantee, and a per-TOKEN rate limit (`shareRL`, 120/min) is the abuse gate (per-IP would see only the proxy). The Next page `/shared/[token]` is account-less (middleware bypass) and fetches via `chatServerFetchPublic` (secret, no user email).
  - **Sweep exemption**: `SweepExpired` skips `share_token IS NOT NULL` rows (TTL delete + per-user cap) so a live share link is never silently revoked by retention.
- **Task Priority Queues (#230)**:
  - **Convention**: `priority` is an integer in `[0, 100]` where **lower = more urgent** (POSIX `nice`-style). The Go zero value (`0`, i.e. unset) is normalized to `models.PriorityNormal` (50) by `models.NewTask`; named reference points live as `models.Priority{Critical,High,Normal,Low,Bulk}` constants. Validated in `validateTaskLimits` and by a DB `CHECK`.
  - **Two columns**: `priority` is the immutable submitted value; `effective_priority` is what the claim path orders by (`ORDER BY effective_priority ASC, created_at ASC`). They are equal at creation. `effective_priority` is **write-once at INSERT** ‚Äî it is deliberately excluded from the `AddTask` upsert/`ON CONFLICT` (and `UpdateTask` delegates to `AddTask`), so a status update with a stale in-memory copy can never clobber a promotion. The ONLY mutator afterward is the anti-starvation sweep.
  - **Anti-starvation**: a per-tick scheduler sweep (`PromoteStarvedTasks`) raises the `effective_priority` of pending tasks waiting past `FLEET_TASK_STARVATION_WINDOW_MINUTES` (default 30; `0` = off) to `StarvationFloorPriority` (High, never Critical), so a stream of urgent work can't starve a low-priority task ‚Äî without rewriting its submitted `priority`.
  - **Per-key cap**: a scoped API key's optional `MaxPriority` (JSON-persisted on the key, not a DB table) makes both `POST /tasks` and `POST /tasks/batch` reject any task more urgent (lower) than the ceiling (`403` / per-task batch failure), via the shared `priorityCapError` helper so the two create paths can't drift.
  - **Inspection**: admin-only `GET /admin/queue` returns per-tier pending depth + oldest-pending age.
  - **Deviation from the issue**: named tiers are Go constants and the window is an env var (matching the existing retention/archival knobs), NOT a `manifest.yaml` map ‚Äî server scheduling policy stays out of the external client bundle.
- **Conversation Folders & Labels (#258)**: builds on the `folder TEXT`/`labels TEXT[]` columns + bulk-mutation from #279.
  - **Filtering**: `GET /conversations?folder=&label=` filters the list ‚Äî `?label=` is repeatable with **AND** semantics (Postgres `labels @> ARRAY[...]`). Backed by `store.ListFiltered(ctx, user, ListFilter{...})`; the old `List(ctx, user, archivedOnly)` now delegates to it. The query is a **constant string with sentinel-guarded clauses** (`$N IS NULL OR ‚Ä¶`, mirroring `DeleteAllMatching`) ‚Äî no SQL concatenation. `folder` is `NOT NULL DEFAULT ''`, so `''` is the explicit "no folder" bucket (a `*string` pointer to `""` filters for it; `nil` = no folder filter).
  - **Enumeration / rename**: `GET /folders` ‚Üí `[{name,count}]` (distinct non-empty folders, active only); `POST /folders/rename {from,to}` is a single bulk UPDATE (no folders table ‚Äî a folder IS the set of conversations naming it). Both scoped by `user_email`.
  - **Validation** (`normalizeAndValidateFolderLabels`, HTTP layer): ‚â§10 labels, ‚â§32 chars each (trimmed, non-empty), folder ‚â§64 chars; applied on the bulk PATCH path ‚Äî this is the **server-side** enforcement of the bounds the UI already applies client-side, closing a gap left by #279's unvalidated bulk mutation. Empty folder (`""`) is allowed = clear.
  - **UI** is owned by #169 (the unified NavRail shell): `conversationOrganization.ts` derives folders/labels **client-side** from the loaded list and renders the organized Pinned¬∑Folders¬∑Labels¬∑Recent list + label chips + kebab folder/label panels. This issue's server-side filtering/enumeration/rename endpoints are API capabilities (matching the #258 spec) the bundled UI doesn't need to call, since it filters client-side.
- **Per-task Sandbox Limits (#205)**: a task may override the global `FLEET_SANDBOX_*` cgroup ceilings.
  - **Model**: `TaskSandboxLimits{MemoryMB, CPUs, Pids}` (a zero field = use the global default), stored as the nullable `sandbox_limits` JSONB column (migration 040; NULL = all-global). Threaded through `db.go` exactly like `loop_config` (taskColumns/scanTask/AddTask/taskInsert\*/UpdateTaskTx) and the export/import record.
  - **Validation** (`Handlers.validateSandboxLimits`): floors (‚â•128 MiB, cpus>0, ‚â•16 pids) and operator ceilings `FLEET_SANDBOX_{MEMORY_MAX_MB,CPUS_MAX,PIDS_MAX}` (defaults 8192/16/1024; 0 = uncapped), surfaced onto `handlers.Config`.
  - **Application**: per-task limits require a COLD start (a warm pooled container already runs with the pool's ceilings). `takeTaskSandbox` routes a task with non-zero `SandboxLimits` through the new `Pool.TakeContainerWithOverrides(ctx, ResourceOverride, noNetwork)` ‚Äî applying the override to a copy of the pool config (`ResourceOverride.applyTo`, the testable seam for the `--memory/--cpus/--pids-limit` values) and keeping the task's network posture (sealed unless `AllowNetwork`). `TakeContainer` now delegates to it. A tightening of the mandatory sandbox, never an escape.
  - **Deferred** (honest scope): the `fleet task set-limits` CLI and the web "Advanced settings" panel ‚Äî per-task limits ship usable now via `POST /tasks` and `fleet task import` (the JSON/YAML carries `sandbox_limits`).
- **Agent delegation (#175 + #264)**: a scheduled run can delegate scoped subtasks to governed CHILD runs via the `spawn_subagent` native tool (this one tool is the delegation primitive #264 asked for ‚Äî NOT a separate `delegate_task`; a second tool would fork the one governed core). See [`docs/adr/0007-governed-sub-agents.md`](docs/adr/0007-governed-sub-agents.md).
  - **Opt-in**: per-task `allow_delegation: true` OR fleet-wide `FLEET_SUBAGENTS_ENABLED` (composed as OR in `scheduledrun`); registered ONLY in scheduled mode, never interactive. Off by default ‚Üí byte-for-byte unchanged. `allow_delegation` is threaded like `allow_network` (migration 041, taskColumns/scanTask/AddTask/taskInsert\*/UpdateTaskTx, export/import, rerun overrides).
  - **Parallel**: the tool is `NewParallelAgentTool`, so multiple `spawn_subagent` calls in one model turn run concurrently (fantasy's parallel-tool semaphore, cap 5) and results fan back in before the next LLM call. Output is JSON `{result, cost_usd, tokens, success}`. The atomic budget reservation (under the parent `mu`) makes concurrent spawns safe; `make test-race` covers it.
  - **Governance**: child runs through the SAME `agentcore.Run`; budget is capped at `FLEET_SUBAGENTS_BUDGET_FRACTION` (default 0.10 = ‚â§10% of the parent's remaining, a request over the cap is REFUSED) and charged back to the parent; `FLEET_SUBAGENTS_MAX_DEPTH` default **1** ("parent ‚Üí sub-agent only" ‚Äî a child is built without the tool); `FLEET_SUBAGENTS_MAX_CHILDREN` default **5** (the 6th spawn returns a "max concurrent sub-agents reached" error result). Optional `timeout_minutes`/`max_iterations` per child; `parent_task_id` links a child's run to its owning task.
- **LLM git-metadata tools (#191)**: three native tools let an agent self-document the code it produces ‚Äî `suggest_branch_name`, `suggest_commit_message`, `suggest_pr_description` (`internal/tools/metadata_tools.go`).
  - **Pattern**: each mirrors `SuggestTitle` (`internal/agent/title.go`) ‚Äî a short-lived `fantasy.NewAgent` call with a tight system prompt and a 20s timeout, resolved through the SAME host-side `ModelResolver` the run already uses. The operator's key stays host-side and never enters the sandbox; the call rides the shared governed provider, not a bare HTTP request.
  - **Validation**: model output is validated before it is returned ‚Äî branch names against `^[a-z][a-z0-9/-]{1,59}$` (plus a guard rejecting `//` and a trailing `/` that the class alone allows but git refuses), commit subjects against the Conventional Commits header + ‚â§72 chars, PR descriptions for the required `## Summary` section. A branch name is thus never an unsafe ref handed to git; the commit *body* and PR description are returned as free-form text/markdown (only their header is validated), which is safe because the agent consumes them inside the mandatory sandbox ‚Äî these tools never interpolate output into a host-side shell. The validators are pure functions (`normalizeBranchName`/`normalizeCommitMessage`/`normalizePRDescription`), unit-tested without a live model.
  - **Config**: `FLEET_METADATA_MODEL` (a `MetadataModel` field on `config.Config`), defaulting to `FLEET_TITLE_MODEL`'s value ‚Äî zero new required config.
  - **Scoped to the scheduled native set**: wired via `tools.MetadataTools(r.mgr, metadataModel)` appended in `internal/scheduledrun`, NOT in the shared `tools.NewTurnTools`/`DefaultTools`. Rationale: autonomous scheduled agents that produce branches/commits/PRs are the use case, and a task's MCP selection is narrow, so the 3 always-on tools stay clear of the 128-tool request ceiling ‚Äî whereas the interactive chat turn runs near that ceiling once per-user remote MCP servers (#449) load, and CI (which uses the 0-MCP default bundle) can't catch an over-128 overflow. `TestMetadataTools_NotInSharedTurnSet` locks this so any future re-addition to the shared set is a deliberate, reviewed decision.
  - **Deviation from the issue** (honest scope): (1) the issue wires these into `DefaultTools()`/`NewTurnTools()` for all native flavors; we scope to scheduled-only for the ceiling reason above (interactive chat does not need an LLM to name a branch ‚Äî a human is present). (2) The issue describes excluding the tools from an external/containment "acp" flavor in `buildFantasyTools`, but no such ACP runtime exists in this codebase. (3) A single `FLEET_METADATA_MODEL` is used (per the issue), not three per-tool model knobs.
- **Client SSE capability negotiation (#194)**: a client can declare which SSE event classes it handles so the server suppresses the rest (`internal/httpapi/capabilities.go`).
  - **Wire protocol**: the client sends `X-Fleet-Capabilities: ["text","tool_calls",‚Ä¶]` (a JSON array) on `POST /chat` and the stream-reattach `GET`; the server replies with `X-Fleet-Supported-Capabilities` and emits a synthetic `fleet.capabilities` event as the **first** SSE frame (no `id:` line, not persisted ‚Äî like `reconnect`/`heartbeat`) so a header-blind `EventSource` can still discover support.
  - **Filtering**: `shouldEmit(caps, name)` gates both the live `turnBuffer.Attach` path and the DB-fallback `replayEventsFromDB`. Only events in `capabilityForEvent` are gated (`text.delta`‚Üí`text`, `reasoning.*`‚Üí`reasoning`, `tool.call`‚Üí`tool_calls`, `tool.result`‚Üí`tool_results`, `tool.approval_*`/`tool.auto_resolved`/`memory.proposed`‚Üí`approval_cards`). **Lifecycle/control frames always flow** (`turn.*`, `conversation*`, `user.message`, `status`, `reconnect`, `buffer_expired`, `fleet.*`) so a filtering client can't break turn-state tracking.
  - **Backward compatible**: no header (or malformed) ‚Üí `parseClientCapabilities` returns `nil` ‚Üí no filter ‚Üí the full stream, exactly as before. Unknown tokens are ignored. The bundled web client sends no header and treats unknown event names as no-ops (chain of `if event.event === ‚Ä¶`), so it is unaffected.
  - **Deviation from the issue** (honest scope): only capabilities for events that ACTUALLY emit here are advertised ‚Äî `text`, `reasoning`, `tool_calls`, `tool_results`, `approval_cards`. The issue's `enforcement_nudges`/`usage_snapshots`/`permissions` are omitted: those event names have **zero** emit sites in this repo (they came from a runtime that doesn't exist here), and advertising a filter that can never fire would be dishonest.
- **Structured output mode (#244)**: a scheduled task can declare an `output_schema` (draft-07 JSON Schema) so its final answer is machine-readable for downstream automation.
  - **Model + storage**: `OutputSchema json.RawMessage` (create input) and `OutputJSON json.RawMessage` (validated result) on `TaskCreate`/`Task`, stored as nullable `output_schema`/`output_json` JSONB (migration 042). Threaded through `db.go` exactly like `result`/`sandbox_limits` (taskColumns/scanTask/AddTask/taskInsert\*/UpdateTaskTx, `marshalRawJSON`/`unmarshalRawJSON`); `output_schema` round-trips on export/import, `output_json` does not (it's a result).
  - **Schema logic** lives in `internal/structuredoutput` (the single source of truth so the create-time gate and post-run validation agree): `ValidateSchema` (create-time compile check, rejecting a malformed schema in `validateTaskCreate`), `PromptAugmentation` (the system-prompt addendum `scheduledrun` appends when a schema is set), and `ValidateOutput` (parses the final answer ‚Äî tolerating a ```json fence or surrounding prose ‚Äî and validates it against the schema). Uses `github.com/santhosh-tekuri/jsonschema/v5`.
  - **Post-run capture**: the agent's final answer lives in the session log (NOT `task.Result`, which holds a status string), so the runner Pool's success path extracts the last assistant message, validates it, and persists `output_json` via a `TaskStatusRunning` status update carrying `OutputJSON` ‚Äî the SAME atomic, lease-checked seam `reportWorkspacePath` (#287) uses ‚Äî BEFORE the terminal success clears the lease. Best-effort: a missing/non-conforming answer leaves `output_json` NULL and the run still succeeds.
  - **Retrieval**: `GET /tasks/{task_id}/output` returns the raw validated JSON (`application/json`, no envelope); `404` when there's no structured output (no schema, or validation failed), `409` while the task hasn't reached a terminal state (`TaskStatus.IsTerminal`).
  - **Deferred** (honest scope): the issue's one-retry-on-invalid-output and task-chaining (`input_from_task_id`) are NOT in this PR. The retry would re-invoke the run, which fights the worker pool's one-claim-one-run model and deserves its own design; chaining is a separate composition feature. The prompt augmentation already steers the model to emit conforming JSON; non-conforming output simply leaves `output_json` NULL (the free-form session log still has the text).
- **Conversation branching (#454)**: fork a chat conversation at a chosen message into a new independent thread.
  - **Model**: nullable `parent_conversation_id`/`branch_point_message_id` columns on `conversations` (chat-store migration 023) ‚Äî lineage metadata only. The branch COPIES the parent's messages (id ‚â§ branch point) into a fresh conversation, so it is fully independent: continuing or deleting either side never affects the other.
  - **Store**: `Store.BranchConversation(user, parentID, branchPointMessageID, title)` loads the parent (user-scoped ‚Üí ownership), copies its messages up to the branch point via `AppendHistory` (so the branch is FTS-indexed like any conversation), inheriting the parent's persona/model/lockdown. A branch point that names no message in the parent returns `ErrBranchPointNotFound`.
  - **API**: `POST /conversations/{id}/branch {branch_point_message_id, title?}` ‚Üí 201 + the new conversation; 404 (not owned) / 400 (missing/unknown branch point). Chat `httpapi` is exempt from the orchestrator OpenAPI parity test, so no `openapi.yaml` entry.
  - **Enabling change**: the conversation GET history now exposes the persisted `messages.id` per entry (`HistoryEntry.ID`, read-only/omitempty) ‚Äî previously the UI only had a client-assigned render index, so a client had no way to name a branch point.
  - **Chat UI** (`web/`): the history reducer (`history.ts`) threads each render-`Message.dbId` = the MAX persisted `messages.id` of the entries aggregated into it (in-flight/streamed messages have none ‚Üí not branchable). `ChatTranscript` renders a "Branch" button on persisted assistant messages; `chat-experience.tsx`'s `branchFromMessage` POSTs to the `/conversations/{id}/branch` Next proxy, then refreshes the sidebar + opens the new branch. Title defaults server-side to "<parent> (branch)".
  - **Deferred** (honest scope): a "forked from X" lineage badge in the sidebar, and branching from a USER message (the button is on assistant messages ‚Äî branching there includes the preceding user turn). The stored `parent_conversation_id` makes both straightforward follow-ups.
- **Typed API keys with scope-based access control (#190)**: keys now carry a legible access class in the token itself ‚Äî `fleet_{type}_{base58}` where `type` ‚àà `admin|task|webhook|readonly` (`internal/sched/apikeys/keytype.go`). The Bitcoin base58 alphabet omits `0/O/I/l` and `+//` so a key is URL- and transcription-safe.
  - **Type is authoritative scope**: the WHOLE raw key (type segment included) is hashed, so a key's wire type can never diverge from its stored `Type` once the hash matches. `ParseAPIKey` classifies a key from its prefix (no store lookup) for a fast format pre-check; enforcement reads the stored `key.Type`. The type DERIVES the permission set (`KeyType.Permissions`: admin‚Üíall, task‚Üícreate/view/logs, readonly‚Üíview/logs, webhook‚Üínone), so typed keys plug into the existing per-handler permission checks unchanged.

  - **Reload precedence (process-env-over-file, like boot ‚Äî for SET values)**: `Load` snapshots the original process-env winners onto `reloadState.bootEnv`; `Reload` re-applies the env file then restores that snapshot, so a value pinned in the process env at boot still wins over the file (re-snapshotting the *current* env would wrongly promote file-only keys and freeze them). Two behaviors DELIBERATELY differ from a fresh boot, both fail-safe (never snap a live value to a default): an **unset/removed** reloadable var keeps the current value (not the built-in default), and a **parse/range failure** is reported in `ReloadResult.Errors` with the previous value KEPT (a bad value never poisons a running process). A changed *non-reloadable* watched setting (effective DB DSN incl. discrete `DB_*` via `buildDatabaseURL`, server addr, auth secrets, max-concurrent-agents, TLS) is reported in `Skipped` with a reason, never silently applied.
  - **Deviation / honest scope**: (1) `FLEET_LOG_LEVEL` is NOT made reloadable ‚Äî fleet has no live log-level config field today (verbosity is consumed ad-hoc), so a reload knob with no sink would be dishonest; it's a follow-on once #178's slog level sink lands. (2) `.env` file-watch auto-reload (`FLEET_ENV_FILE_WATCH` via fsnotify) is deferred ‚Äî it would add a filesystem-watch dependency, and the signal + HTTP triggers fully deliver reload (drive them from a config-management tool / deploy hook). (3) `handlers.Config`'s boot-copied `MaxCostUSD` (used only by the `/tasks/estimate` forecast) is intentionally NOT live ‚Äî it's a cost *estimate*, not an enforced ceiling. (4) No security-boundary change (no ADR): auth secrets and DB DSNs are explicitly non-reloadable.
- **Self-hosted eval & regression harness (#502)**: replay "golden" prompts through the governed loop and gate model swaps / bundle edits on regressions. See [`docs/EVALS.md`](docs/EVALS.md) + [`docs/adr/0018-self-hosted-eval-harness.md`](docs/adr/0018-self-hosted-eval-harness.md).
  - **Goldens are client content**: eval sets live in the bundle's `evals/` dir (one YAML per set: `set`/`threshold`/`judge_model`/`cases[]`), loaded by `internal/evals.LoadSets` with the skills-loader contract (absent dir fine; malformed set EXCLUDED + reported, never half-run). `clientconfig.Bundle` only gains `EvalsDir` ‚Äî no manifest key, no Load-time validation (clientconfig cannot import evals: cycle via agent). The generic bundle ships `config/default/evals/example.yaml` (a template; `TestDefaultBundleEvalsValid` keeps it clean).
  - **Replay = the ONE governed loop**: `fleet eval run <set>` builds the same `agent.Manager` the server boots (no MCP specs ‚Äî native tools only, honest scope) and drives each case through `Manager.RunTurn ‚Üí agentcore.Run` at the case's PINNED model; persona/prompt CONTENT deliberately resolves from the live bundle (a bundle edit is what the eval detects). Cases run sequentially, each in its own `eval-<runid>-<n>` conversation workspace; replay temperature is pinned via `--temperature` (default 0) ‚Äî safe direct Config field write, the CLI registers no reload triggers.
  - **Scorers**: deterministic ones extracted to the shared `internal/scorers` (Regex/Shell/Contains/Equals/FirstWordIsYes/LastAssistantMessage ‚Äî `scheduledrun/loop.go` now delegates; the `TaskIteration.ExitConditionResult` label contract preserved verbatim, and the nil-sandbox guard stays in loop.go since a typed-nil `*sandbox.Sandbox` in the `scorers.BashRunner` interface would defeat the nil check). The LLM-judge (`internal/evals/judge.go`) is the AnalyzeTaskFailure pattern: host-side resolver, temperature 0, fixed `{pass,score,reasoning}` draft-07 schema via `structuredoutput`, ONE corrective retry feeding the validation error back (the #502-noted no-retry-gap handling), judge model = scorer.model ‚Üí set.judge_model ‚Üí case.model. Judge ERROR = scorer FAILS (`judge:error`, fail-closed). Case passes iff ALL scorers pass; set passes iff pass-fraction ‚â• threshold (default 1.0) ‚Äî that's the exit-0/1 CI gate.
  - **Persistence**: sched migration 046 `eval_runs` (immutable rows: aggregate + per-case JSONB + `bundle_sha` from `evals.BundleFingerprint` ‚Äî sha256 over manifest+prompt/persona/protocol/skill/eval RAW bytes, pre-interpolation so env secrets can't influence it; mcp/ + sandbox/ deliberately excluded). Accessors `AddEvalRun`/`ListEvalRuns`/`LatestEvalRun` (`internal/sched/db/evalruns.go`); persistence is best-effort (`--no-db` / missing DSN = warning, the gate still works ‚Äî bundle-repo CI has no orchestrator DB).
  - **CLI**: `fleet eval run|list|history|capture` ‚Äî a server-family verb (`cmd/fleet/eval.go`, validate-config pattern; exit 2 usage / 1 fail / 0 pass). `capture --task <uuid>` (prompt+model+persona from the task, `expected` from its LATEST log ‚Äî the logs table upserts per task) or `--conversation <id> --user <email>` (FIRST user text ‚Üí prompt, last assistant ‚Üí expected), appends to `<bundle>/evals/<set>.yaml` with a default llm_judge scorer to review.
  - **Deviation / honest scope**: no MCP tools in replays; no online scoring/canary; no replay cache (fingerprint recorded, not yet a cache key); no web UI; no per-case temperature/network/iterations; multi-turn chat captures replay one-shot. NO new HTTP routes (openapi.yaml untouched).
